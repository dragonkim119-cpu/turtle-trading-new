## 1. 목표

실전 사용 가능한 터틀 트레이딩 시스템. 바이낸스 선물(암호화폐) 대상으로 돈치안 돌파
신호를 자동 감시하고, 텔레그램으로 매수/매도/손절/트레일링 알림을 발송하며,
모바일 웹에서 차트·신호·포지션·뉴스를 확인한다.

v2 추가 목표: 포트폴리오 수준 리스크 통제(총 익스포저 캡, 손실 스로틀)와,
향후 메타 레이블링(Phase 2 ML)을 위한 신호 시점 피처 데이터 축적.

근거 문서 (프로젝트 루트 3개):
- [실전 지침서] 터틀 트레이딩 원칙 기반의 체계적 매매 운용 가이드
- 정량적 리스크 관리 전략서: ATR 기반 변동성 통제와 수익 구조의 안정화
- [입문자 가이드] 데이터로 증명된 승리: 터틀 트레이딩 시스템과 프라이스 액션

| 항목 | 결정 |

| 차트 | TradingView Lightweight Charts (오픈소스) |
| 타임프레임 | 일봉(1d) + 4시간봉(4h), 심볼·타임프레임별 독립 설정 |
| 감시 종목 | 웹에서 워치리스트 직접 관리 (기본값: BTCUSDT, ETHUSDT) |
| 운영 환경 | 클라우드 (Railway 또는 동급), web + engine 단일 프로젝트 배포 |
| 포지션 관리 | 수동 등록 → 등록된 포지션 기준 손절/트레일링/청산 알림 |
| 뉴스/캘린더 | 경제 캘린더 = TradingView 위젯, 뉴스·속보 = 서버 RSS 수집 + 키워드 필터 |

| **ML 정책 (v2)** | v1 범위에 ML 없음. 단, Phase 2 메타 레이블링 대비 **신호 시점 피처 스냅샷을 지금부터 DB에 축적** (§8). 가격 방향 예측 모델은 영구 배제 |

## 3. 전체 구조

```
turtle-trading/  (TypeScript 모노레포, pnpm workspace)
├─ apps/
│  ├─ web/        Next.js 웹앱 (모바일 우선 반응형)
│  └─ engine/     신호 엔진 워커 (24시간 상주 프로세스)
├─ packages/
│  └─ core/       공유 패키지: 지표 계산, 신호 판정, 사이징, 포트폴리오 리스크, 타입
├─ data/          SQLite DB 파일
└─ docs/
```

- **packages/core가 단일 진실 공급원.** 웹 차트 지표 값과 텔레그램 신호 값이 동일 함수에서 나온다.
- DB: SQLite (better-sqlite3). 단일 사용자, 파일 백업 용이.
- 배포: Railway 단일 프로젝트에 web / engine 두 서비스, 볼륨에 SQLite 저장.

## 4. 트레이딩 알고리즘

### 4.1 기본 터틀 규칙 (v1과 동일)

파라미터 (심볼·타임프레임별 설정 가능, 아래는 기본값):
- 돈치안 진입 채널: 20봉 / 청산 채널: 10봉
- ATR 기간: 20, 손절 배수: 2.0
- 추세 필터: 200 EMA
- 리스크: 계좌 대비 거래당 2%

판정은 전부 **봉 마감 확정 종가** 기준.

| 이벤트 | 조건 |
|---|---|
| 롱 진입 | 종가 > 200 EMA AND 종가가 직전 20봉 최고가 돌파 AND 보완 필터 통과 |
| 숏 진입 | 종가 < 200 EMA AND 종가가 직전 20봉 최저가 이탈 AND 보완 필터 통과 |
| 초기 손절 | 롱: 진입가 − 2×ATR(진입 시점) / 숏: 진입가 + 2×ATR |
| 트레일링 | 10봉 반대선이 기존 스톱보다 유리해지면 스톱 갱신 |
| 청산 | 롱: 종가가 10봉 최저가 이탈 / 숏: 종가가 10봉 최고가 돌파 |
| 손절 집행 | 보유 포지션 마크가격이 손절선 도달 — 장중 1분 폴링 감시 |

포지션 사이징 (2% 룰): `수량 = (계좌자산 × 리스크%) ÷ (손절폭 = 2×ATR)`

### 4.2 보완 필터 (v1 4종 + v2 1종 추가)

각 필터는 심볼·타임프레임별 on/off + 임계값 설정 가능. 진입 = (기본 터틀 조건) AND
(활성화된 모든 필터 통과). 청산·손절에는 필터 미적용.

| # | 필터 | 기본값 | 상태 | 로직 |
|---|---|---|---|---|
| 1 | ADX 추세강도 | 기간 14, 임계 20 | 기본 on | ADX ≥ 20일 때만 진입. 횡보장 돌파 차단 |
| 2 | 거래량 확인 | 20봉 평균 × 1.5 | 기본 on | 돌파봉 거래량 ≥ 평균×배수 |
| 3 | Rolling VWAP | 30일 | 기본 on | 롱: 종가 > VWAP / 숏: 종가 < VWAP |
| 4 | 펀딩비 과열 | ±0.1% | 기본 on | 과열 방향 진입 보류. 과밀 청산 위험 회피 |
| **5** | **OI 확인 (v2 신규)** | 24h OI 변화율 > 0% | **기본 off** | 돌파 방향 + 미결제약정 증가 = 신규 자금 유입(진짜 돌파). OI 감소 중 돌파 = 숏커버/롱스퀴즈 가능성 → 보류. 데이터: 바이낸스 `/futures/data/openInterestHist` (무료). **백테스트로 유효성 확인 후 on 전환 판단** |

주: 필터 추가는 5번에서 동결. 필터 수 늘수록 거래 표본이 줄어 통계적 우위 자체가
붕괴한다(외부 제안의 직렬 6게이트 구조가 실패하는 이유). 신규 필터 후보는 반드시
백테스트 CLI의 조합 비교를 통과해야 채택.

### 4.3 청산 옵션 2종 (v2 신규 — 기본 off, 백테스트로 채택 결정)

원조 터틀 청산(10봉 반대선)의 이익 반납 문제 보완 후보. 켜기 전 반드시 백테스트
비교표에서 기본 청산 대비 우위 확인.

| 옵션 | 기본값 | 로직 |
|---|---|---|
| 부분익절 | off / 2R에서 1/3, 4R에서 1/3 | 도달 시 텔레그램 부분익절 권고 알림. 잔여 물량은 기존 트레일링 유지 |
| 타임 스톱 | off / 진입 후 12봉 | N봉 내 +1R 미도달 시 청산 권고 알림. 횡보 물림 제거 |

### 4.4 포트폴리오 리스크 규칙 (v2 신규)

심볼 단위 2% 룰만으로는 BTC·ETH 동시 포지션 = 사실상 한 방향 4% 베팅 (상관 ~0.8+).
계좌 수준 통제 추가. 알림 전용 시스템이므로 "집행"이 아니라 **신호 억제 + 경고**로 구현.

| 규칙 | 기본값 | 동작 |
|---|---|---|
| 동시 오픈 리스크 총량 캡 | 6% (= 2% × 3포지션) | 등록 포지션들의 (진입가−현재스톱)×수량 합산이 캡 초과 시, 신규 진입 신호를 "정보성"으로 강등 발송 ("신호 발생. 단, 오픈 리스크 6.0%/6.0% — 신규 진입 비권장") |
| 방향 편중 경고 | 동일 방향 3개 | 같은 방향 포지션 3개 이상이면 신호에 편중 경고 문구 첨부 |
| 손실 스로틀 (일) | 등록 포지션 합산 −4%/일 | 당일 실현+미실현 손실 초과 시 당일 잔여 신규 진입 신호 전부 정보성 강등 + 경고 알림 |
| 손실 스로틀 (월) | −10%/월 | 월 누적 초과 시 "리스크 절반 권고" 알림 + 사이징 계산기가 리스크 1%로 자동 표시 |

모든 임계값은 설정에서 수정 가능. 스로틀 상태는 웹 대시보드에 배지로 상시 표시.

### 4.5 신호 상태 머신 (v1과 동일)

`FLAT → LONG_SIGNALED → LONG_OPEN → (트레일링 갱신...) → 청산/손절 → FLAT` (숏 대칭).
- LONG_SIGNALED: 진입 신호 발송됨, 포지션 등록 대기
- LONG_OPEN: 포지션 등록 완료 → 손절 실시간 감시 + 트레일링/청산 알림 활성
- 포지션 미등록 상태에서도 청산 신호는 참고용 발송

## 5. 신호 엔진 (apps/engine)

- 바이낸스 선물 공개 REST API: `/fapi/v1/klines`, `/fapi/v1/premiumIndex`,
  `/futures/data/openInterestHist` (v2 추가). 전부 무료·API 키 불필요.
- 매크로 표시 데이터 (v2 추가): DXY·VIX·미 10Y 금리 — FRED/Stooq 무료 소스, 1일 1회
  수집. **표시 전용, 신호 게이트에 사용하지 않음** (게이트화하려면 백테스트 검증 선행).
- 스케줄:
  - 봉 마감 감지: 4h/1d 마감 + 30초 지연 후 확정 캔들 조회 → 신호 판정
  - 손절 감시: 등록 포지션 있을 때만 1분 간격 마크가격 폴링
  - OI/펀딩 스냅샷: 봉 마감 판정 시점에 함께 수집
  - RSS 수집: 10분 간격 / 매크로 지표: 1일 1회
- 중복 발송 방지: 유니크 키 = (심볼, 타임프레임, 이벤트, 봉 시각). DB unique 제약.
- 재시작 복구: 마지막 처리 봉 시각 기록, 기동 시 누락 봉 소급 판정.
- 자체 경보: API 연속 실패(5회), 텔레그램 발송 실패 누적 시 엔진 이상 경보.

## 6. 웹앱 (apps/web)

모바일 우선 반응형. 하단 탭 4개. (외부 제안의 네이티브 앱은 배제 — 모바일 웹 +
텔레그램이 "조종석" 역할을 1/100 비용으로 수행. §13 참조)

### 6.1 차트 탭
- Lightweight Charts: 캔들 + 돈치안 진입/청산 채널, 200 EMA, Rolling VWAP,
  과거 신호 마커(▲▼✕), 보유 포지션 진입가·손절선·트레일링 이력
- 심볼·타임프레임 전환, 설정 시트에서 파라미터·필터 수정
- 하단 정보 바: ATR, ADX, 펀딩비, OI 24h 변화율(v2), 돌파까지 거리 %

### 6.2 신호 탭
- 신호 히스토리 (필터 차단 이력 포함)
- 심볼별 현재 상태 카드: 상태 머신, 돌파 근접도, 필터 통과 현황
- **포트폴리오 리스크 카드 (v2)**: 오픈 리스크 합계/캡, 방향 편중, 스로틀 상태

### 6.3 포지션 탭
- 신호 기반 포지션 등록: 신호 선택 → 실제 진입가·수량 입력
- 2% 계산기 (스로틀 발동 시 자동 1% 표시)
- 보유 포지션 목록: 미실현 손익, 현재 스톱, 청산 조건 거리, R 배수 진행도(v2 — 부분익절 옵션용)

### 6.4 뉴스 탭
- 경제 캘린더: TradingView Economic Calendar 위젯
- 뉴스 피드: 서버 수집 RSS (소스 목록 설정 파일 관리)
- 속보 하이라이트: 키워드 매치 상단 고정 + 배지, 텔레그램 발송 옵션
- **매크로 카드 (v2)**: DXY·VIX·10Y 금리 현재값 + 최근 추이 스파크라인 (정보 표시 전용)

### 6.5 설정
- 계좌자산, 리스크%, 워치리스트
- 심볼·타임프레임별 파라미터/필터/청산 옵션
- 포트폴리오 리스크 규칙 임계값 (v2)
- 텔레그램 알림 종류별 on/off
- 웹 접근 보호: 단일 비밀번호 (환경변수, 세션 쿠키)

## 7. 텔레그램 메시지 형식

```
🟢 롱 진입 신호 — BTCUSDT · 4H
진입가: 65,000
손절가: 63,560 (−2×ATR, ATR20=720)
권장수량: 0.138 BTC (시드 ₩100,000,000 · 리스크 2%)
필터: ADX 28 ✅ | 거래량 2.1× ✅ | VWAP 상방 ✅ | 펀딩 +0.03% ✅ | OI +2.4% ✅
포트폴리오: 오픈 리스크 2.0%/6.0% · 편중 없음
```

- 트레일링: `🔁 스톱 상향 — BTCUSDT · 4H | 63,560 → 64,900`
- 청산: `🔴 청산 신호 — 10봉 저점 이탈 종가 마감`
- 손절: `⛔ 손절선 도달 — 즉시 청산 (마크가격 63,540 ≤ 63,560)`
- 정보성 강등 (v2): `⚠️ 신호 발생 — 진입 비권장 | 사유: 오픈 리스크 캡 도달 (6.0%/6.0%)`
- 부분익절 (옵션): `💰 2R 도달 — 1/3 익절 권고 | 잔여 트레일링 유지`

주의: 메시지에 "승률", "확신도", "점수" 류의 수치는 표기하지 않는다. 검증된
캘리브레이션 없는 확률 표시는 과잉신뢰만 유발 (외부 제안 폐기 사유, §13).

## 8. DB 스키마 (SQLite, 개요)

- `symbols`, `settings`, `params` (v1과 동일 + 청산 옵션·OI 필터 설정 추가)
- `signals`: 신호 이력, 유니크 키, 필터 스냅샷 JSON
  - **`feature_snapshot` JSON (v2 신규)**: 신호 시점의 전체 지표 상태 — ATR, ADX,
    거래량 배수, VWAP 이격률, 펀딩비, OI 변화율, 200EMA 이격률, 돌파 강도(종가−채널선
    /ATR), 요일·세션. **목적: Phase 2 메타 레이블링 학습 데이터.** 지금 안 쌓으면
    나중에 소급 불가 — v2에서 가장 저렴하고 가치 높은 추가 항목
- `positions`: 수동 등록 포지션, 상태, 스톱 이력 + **결과 R 배수 기록 (v2)** —
  청산 시 실현 R을 기록해 신호↔결과 레이블 완성
- `portfolio_state` (v2): 일/월 누적 손익, 스로틀 상태
- `macro_snapshots` (v2): DXY·VIX·금리 일별 값
- `news_items`, `engine_state` (v1과 동일)

## 9. 에러 처리 (v1과 동일 + OI/매크로 소스 독립 실패 허용)

- 바이낸스 API: 지수 백오프, rate limit 준수
- 캔들 무결성: 봉 시각 연속성 검증, 갭 재조회
- OI·매크로 데이터 실패: 해당 필터/카드만 "데이터 없음" 처리, 나머지 신호 판정은 정상
  진행 (OI 필터 on인데 데이터 실패 시 → 필터 통과로 간주하지 않고 "판정 불가" 표기 후
  정보성 신호로 강등)
- 텔레그램: 재시도 3회, 실패 시 `delivery_failed` 마킹
- RSS: 소스별 독립 실패 허용

## 10. 테스트·검증

- packages/core 유닛 테스트: 지표 계산 기대값 대조, 신호 시나리오 테스트
- **백테스트 CLI** (`pnpm backtest`) — v2에서 역할 격상. 모든 선택 항목의 채택 게이트:
  - 필터 조합별 비교 (특히 OI 필터 on/off), 청산 옵션 조합별 비교
  - **수수료(taker 0.05% 왕복) + 슬리피지(0.05%) 반영 필수** — 미반영 백테스트 결과는 무효
  - 출력: 거래 수, 승률, 손익비, Profit Factor, MDD, 연도별 분해 (2021 강세·2022 약세
    구간 분리 확인)
  - 단순 in-sample 최적화 금지: 파라미터 탐색 시 기간 분할 (예: ~2023 탐색 / 2024~ 검증)
- 실전 투입 전 1~2주 신호 관찰 운용

### 성과 기대치 (명시)

목표 KPI를 낙관 수치로 박아두지 않는다 (외부 제안의 CAGR 40~80%·Sharpe 2+는 판타지).
현실 기준선: 추세추종 승률 40~45%, 손익비 2+, MDD 25~35% 각오. 백테스트에서
승률 60%·PF 3.5 같은 수치가 나오면 성공이 아니라 **과최적화 경보**로 취급한다.

## 11. 범위 제외 (v1에서 안 함)

- 자동 주문 집행 (거래소 API 키 연동 매매) — 알림 전용
- 거래소 잔고 자동 연동
- 다중 사용자 지원
- 한국·미국 주식 (데이터 소스 인터페이스만 분리 설계)
- ML 모델 (Phase 2 — 아래 §12 참조)

## 12. 구현 순서

1. 모노레포 스캐폴드 + packages/core 지표·신호·포트폴리오 리스크 로직 + 유닛 테스트
2. 백테스트 CLI — 기본 터틀 + 필터 4종 검증, OI 필터·청산 옵션 채택 여부 결정
3. engine: 캔들·펀딩·OI 수집 → 신호 판정 → 피처 스냅샷 저장 → 텔레그램 발송
4. web: 차트 → 신호/포지션 → 뉴스/매크로 → 설정
5. 배포 (Railway) + 1~2주 관찰 운용
6. (Phase 2, 신호 이력 6개월+ 축적 후) 메타 레이블링: 축적된 feature_snapshot + 실현 R
   레이블로 XGBoost 이진 분류기 학습 — "이 신호가 +1R 이상 갈 확률". 신호 차단이 아니라
   알림에 참고 정보로 첫 도입, purged K-fold 검증 통과 시에만. 가격 방향 예측 모델은
   Phase 2에서도 하지 않는다
7. (Phase 2, 선택) LLM 일일 브리핑: 당일 신호·포지션·뉴스 요약 텔레그램 발송.
   매매 판단에는 LLM 미사용

## 13. 검토 후 폐기한 외부 제안 (기록용)

| 제안 | 폐기 사유 |
|---|---|
| 직렬 6게이트 파이프라인 (Macro→Regime→...→Score) | 결합 통과율 급락 → 연 거래 3~5건, 통계적 우위 성립 불가 |
| AI Scoring (수기 가중치 Macro 25점 등) | 근거 없는 가중치 = 손으로 한 곡선맞춤 |
| 가격 방향 예측 ML (3/5/10일 상승확률) | 신호대잡음비 최악, 과최적화 1순위. 메타 레이블링으로 대체 |
| 온체인 유료 데이터 (MVRV, 고래 지갑 등) | 유료 + 일봉 지연 + 4h 전략에 부적합. 비용 대비 기여 미검증 |
| 옵션 Greeks (Gamma, Max Pain, IV) | BTC/ETH 외 유동성 없음, 데이터 접근 비용 과다 |
| Confidence %·별점·Strong Buy 라벨 | 캘리브레이션 없는 확신도 표시 = 과잉신뢰 유발 |
| 네이티브 모바일 앱 | 모바일 웹 + 텔레그램이 동일 역할. 검증 안 된 전략에 앱 선투자 금지 |
| Kafka·Kubernetes·TimescaleDB | 단일 사용자 4h/1d 전략에 과잉. SQLite + 단일 VPS로 충분 |
| HMM 레짐 분류 | ADX 필터가 동일 목적을 검증 가능한 형태로 수행. 복잡도 대비 이득 미입증 — Phase 2 이후 백테스트 게이트 통과 시에만 재고 |
| 낙관 KPI (CAGR 40~80%, Sharpe 2+) | 달성 사례 없음. §10의 현실 기준선으로 대체 |

상세 구현 계획은 별도 문서로 작성 (writing-plans 단계).
